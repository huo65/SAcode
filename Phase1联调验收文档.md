## 1. 验收范围

本阶段只验证当前已经落地的核心链路，不引入新的业务需求，重点覆盖：

- 数据库初始化可成功完成。
- 前后端可在本地正常启动并联通。
- 顾客、商家、骑手、管理员四类角色都能完成最小操作。
- 订单主流程可从商品上架走到顾客收货、评价、商家回复。

说明：

## 2. 环境前置

### 2.1 建议环境

- JDK `1.8`
- Maven 可用本地安装版
- Node.js `18.x`
- npm `10.x`
- MySQL `8.x` 或兼容版本

### 2.2 关键目录

- 后端：`backend/DB-market`
- 前端：`front`
- 初始化脚本：`backend/DB-market/src/main/resources/schema.sql`
- 后端配置模板：`backend/DB-market/src/main/resources/application.example.yml`

## 3. 初始化与启动步骤

### 3.1 初始化数据库

1. 创建或连接本地 MySQL。
2. 执行以下脚本：

```sql
source src/main/resources/schema.sql;
```

脚本会完成以下内容：

- 创建数据库 `market`
- 创建用户、商品、订单、购物车、评价等核心表
- 插入演示账号
- 插入基础分类与地址

### 3.2 配置后端

在 `backend/DB-market/src/main/resources/` 下准备本地 `application.yml`，可基于 `application.example.yml` 复制后修改，至少保证以下配置正确：

- `spring.datasource.url`
- `spring.datasource.username`
- `spring.datasource.password`
- `alipay.*` 可先使用本地占位配置，Phase 1 不要求完成真实支付回调联调

补充说明：

- 默认后端端口为 `8080`
- JWT 密钥可通过环境变量 `DBMARKET_JWT_SECRET` 提供；未提供时可使用开发回退值

### 3.3 启动后端

在 `backend/DB-market` 目录执行：

```powershell
mvn test
```

通过后可使用以下任一方式启动：

```powershell
mvn spring-boot:run
```

或直接通过 IDE 启动 `DBmarketApplication`

### 3.4 启动前端

在 `front` 目录执行：

```powershell
npm install
npm run dev
```

默认前端会请求 `//127.0.0.1:8080`，因此本地联调时需要保证后端运行在 `8080` 端口。

### 3.5 构建校验

建议在开始联调前执行：

```powershell
cd backend/DB-market
mvn test

cd ../../front
npm run build
```

补充建议：

- 可结合 `AGENTS.md` 中“联调经验与注意事项”执行，遵循“先编译，再联调”“启动成功不等于可用”“按页面/上传/支付分层验证”的原则。
- 若后续启动方式或建议命令发生变化，应更新本文件；若该变化同时影响项目级通用联调建议，再同步更新 `AGENTS.md`，若影响当前阶段验证状态，再同步更新 `功能完善进度.md`。

## 5. 四角色最小联调主流程

推荐按以下顺序进行一次完整联调，以保证链路连续。

### 5.1 步骤 A: 商家新增商品

登录角色：`merchant`

操作目标：

- 进入商品管理页
- 新增一个商品
- 填写商品名称、分类、价格、库存、描述
- 保存成功后确认商品进入待审核状态

验收要点：

- 商家只能看到并操作自己的商品
- 新增商品后不会直接对顾客可见

### 5.2 步骤 B: 管理员审核商品

登录角色：`admin`

操作目标：

- 进入商品管理页
- 找到商家刚新增的商品
- 执行审核通过

验收要点：

- 管理员可以看到待审核商品
- 审核通过后，顾客端可检索该商品

### 5.3 步骤 C: 顾客浏览餐厅并下单支付

登录角色：`customer`

操作目标：

- 进入首页餐厅列表
- 打开餐厅详情
- 找到已审核商品并加入购物流程
- 选择地址
- 填写订单备注、期望送达时间
- 提交订单并完成支付

验收要点：

- 顾客首页为餐厅维度浏览入口
- 餐厅详情可查看菜单
- 创建订单时后端负责金额计算
- 支付成功后订单进入 `已支付/待商家处理`

### 5.4 步骤 D: 商家接单并完成备餐

登录角色：`merchant`

操作目标：

- 进入订单页
- 看到新支付订单
- 执行接单
- 将订单流转到备餐中
- 备餐完成后推送给骑手

验收要点：

- 新订单提醒可见
- 商家只能处理自己的订单
- 状态流转应为 `0 -> 4 -> 3`

### 5.5 步骤 E: 骑手接单并配送

登录角色：`driver`

操作目标：

- 进入订单页
- 找到待接单订单
- 执行接单
- 完成配送

验收要点：

- 骑手只能看待接单订单或分配给自己的订单
- 状态流转应为 `3 -> 1 -> 2`

### 5.6 步骤 F: 顾客收货并评价

登录角色：`customer`

操作目标：

- 打开订单页
- 对已送达订单确认收货
- 对已完成订单进行评分和文字评价

验收要点：

- 只有已完成订单允许评价
- 同一订单只能评价一次
- 订单列表中可看到评价状态

### 5.7 步骤 G: 商家回复评价

登录角色：`merchant`

操作目标：

- 打开订单页
- 找到已被评价的订单
- 执行回复评价

验收要点：

- 只有对应商家可以回复
- 同一评价只能回复一次
- 顾客在餐厅详情中可看到回复内容

## 7. 建议补充回归分支

完成主流程后，建议至少补测以下异常或补充分支：

### 7.1 商家拒单退款

- 顾客创建并支付订单
- 商家执行拒单并填写原因
- 验证订单变为退款完成状态
- 验证顾客余额回滚
- 验证商品库存恢复

### 7.2 商家取消备餐中订单

- 顾客支付订单
- 商家接单并进入备餐中
- 商家取消订单并填写原因
- 验证余额与库存均回滚

### 7.3 顾客申请退款

- 顾客支付后申请退款
- 商家或管理员确认退款
- 验证退款状态与退款原因展示

### 7.4 匿名浏览安全

- 未登录状态访问商品搜索、分类、餐厅列表、餐厅详情
- 验证匿名只能看到公开数据
