# AGENTS.md

## 项目目标

本项目后续开发与修复工作的总目标，是根据文档 `c:\Users\huozj\Desktop\SA\SAcode\待完善功能.md` 持续完善当前外卖系统原型，使其尽可能贴近课程材料中的完整业务闭环。

同时需要结合以下两份项目文档协同推进：

- `c:\Users\huozj\Desktop\SA\SAcode\功能完善进度.md`：用于记录各模块当前完成状态、阶段路线图、验证状态和后续待办。
- `c:\Users\huozj\Desktop\SA\SAcode\Phase1联调验收文档.md`：用于记录 Phase 1 联调初始化步骤、四角色最小主流程、操作清单和验收记录。

本项目的使用场景以课堂展示和课程验收为主，因此开发取向应遵循以下原则：

- 优先保证功能完整、流程可演示、页面可操作、角色链路闭环。
- 优先补齐顾客、商家、骑手、管理员四类角色的核心业务路径。
- 优先实现“看得见、能演示、可讲解”的能力，不把主要精力消耗在高并发、分布式、极限性能等展示价值较低的细节上。
- 非功能建设以“足够支撑课堂演示”为标准，避免过度设计。

## 开发总原则

### 1. 以 `待完善功能.md` 为唯一任务基线

- 新增功能、修复问题、补文档时，首先对照 `待完善功能.md`。
- 若实际代码现状与文档不一致，应优先更新代码以贴近文档中的目标方向。
- 如需取舍，优先完成文档中 `P0`、`P1` 项，再考虑 `P2`。

### 1.1 文档协同维护原则

- `待完善功能.md` 负责定义“要做什么”。
- `AGENTS.md` 负责定义“按什么目标和原则来做”。
- `功能完善进度.md` 负责记录“做到什么程度了”。
- `Phase1联调验收文档.md` 负责记录“Phase 1 如何初始化、如何联调、如何验收”。
- 后续更新时，应按职责范围更新对应文档，而不是默认三份文档一起修改。
- 若改动只影响项目目标、开发原则、取舍边界或通用联调方法，更新 `AGENTS.md` 即可。
- 若改动只影响模块完成状态、阶段路线图、验证状态或待办列表，更新 `功能完善进度.md` 即可。
- 若改动只影响 Phase 1 的初始化步骤、联调流程、操作清单、验收记录或当前结论，更新 `Phase1联调验收文档.md` 即可。
- 只有当一次改动同时跨越多个职责范围时，才需要同步更新对应文档。
- 若 Phase 1 联调范围、验收步骤或结论发生变化，至少需要回填 `功能完善进度.md` 中相关的验证状态、阶段路线图或验收记录；若变化进一步影响通用规则，再更新 `AGENTS.md`。

### 2. 以“可演示”优先于“工程完美”

- 优先做完整主流程，而不是过早追求架构重构。
- 优先保证页面、接口、数据库流程能跑通，再考虑抽象优化。
- 对课堂展示帮助不大的复杂并发处理、分布式扩展、极限压测、复杂容灾方案，不作为当前阶段重点。
- 可以保留后续扩展接口，但当前实现应先服务于本地演示与联调。

### 3. 功能闭环优先于边角完备

- 一个模块至少要具备“入口、操作、结果反馈、状态变化”四要素。
- 需要优先打通的不是零散按钮，而是完整链路，例如：
  - 顾客浏览 -> 下单 -> 支付 -> 收货
  - 商家上架 -> 审核 -> 接单 -> 发货
  - 骑手接单 -> 配送 -> 完成
  - 管理员审核 -> 管理 -> 统计展示

### 4. 本地可运行优先

- 所有接入外部服务的能力，优先保证本地可演示替代方案。
- 若对象存储、支付、短信、地图等外部能力暂未完全打通，应优先准备本地可运行方案，不阻塞主流程展示。
- 替代方案应尽量保留统一接口，方便后续无缝切换到真实服务。

## 当前阶段优先级

依据 `待完善功能.md`，当前阶段建议按以下顺序推进：

### P0：先保证主链路能演示

- 认证授权基础闭环：至少保证登录态、角色区分、基础权限可用。
- 订单主流程：下单、支付、状态流转、退款主链路可演示。
- 骑手端核心页面：至少能登录、看单、接单、更新配送状态。
- 敏感配置整理：避免把明显密钥硬编码在业务代码里。

### P1：补齐课堂展示效果

- 顾客端餐厅化展示、搜索、备注、送达时间、评价。
- 商家端接单、拒单、备餐、销售统计。
- 管理端用户/商家/骑手管理与基础报表。
- 支付退款状态同步、图片上传、页面完整度优化。

### P2：适度补非功能

- 分页、基础缓存、错误提示、文档整理。
- 基础测试、健康检查、数据备份说明。
- 响应式适配、国际化、兼容性等。

说明：

- `P2` 不是不做，而是不能挤占 `P0/P1` 主流程时间。
- 若课堂展示时间紧，应优先保证“讲得通、演得出、角色链路完整”。

## 具体实施要求

### 1. 页面与交互

- 页面允许简化，但不能缺少关键演示入口。
- 优先消除空白页、角色页加载失败、按钮无响应、核心数据不显示等问题。
- 对关键页面应补充空状态、成功提示、失败提示，保证演示时可解释。

### 2. 后端与接口

- 接口优先追求稳定、返回一致、能支撑前端展示。
- 重要业务操作应尽量由后端决定状态，不依赖前端硬编码结果。
- 若真实业务规则太复杂，可采用“课堂展示版”简化实现，但要保证逻辑自洽。

### 3. 数据与配置

- 优先使用本地可复现的演示数据。
- 所有课堂演示依赖的账号、配置、启动方式，应尽量文档化。
- 能放到配置文件中的参数不要散落在代码中。

### 4. 非功能边界

当前阶段不应投入过多精力的方向包括：

- 复杂并发一致性处理
- 大规模高并发压测
- 分布式拆分与微服务改造
- 复杂消息队列补偿体系
- 面向生产环境的高可用灾备体系

当前阶段应做到“基本合理”即可的方向包括：

- 简单权限控制
- 基础错误处理
- 演示级监控与日志
- 基础测试与启动校验

## 联调经验与注意事项

### 1. 先编译，再联调

- 先跑后端测试或编译，再跑前端构建，能提前暴露明显错误。
- 推荐顺序：
  1. 后端 `mvn "-Dmaven.repo.local=.m2" test`
  2. 前端 `npm run build`
  3. 再分别启动后端和前端进行联调

### 2. 启动成功不等于可用

- 不能只看命令行显示服务启动，还要实际访问页面或接口确认可用。
- 前端要检查页面是否真的加载、路由是否正常、浏览器是否不是空白页。
- 后端要检查端口是否可达，接口是否能正常返回。

### 3. 前端启动参数尽量简单

- 实测前端直接使用 `npx vite --host 127.0.0.1 --port 5175` 更稳定。
- 某些 `npm run dev -- --host ...` 写法可能出现“进程在跑但页面未正常挂起”的情况。
- 联调时应固定一个明确地址，例如：
  - `http://127.0.0.1:5175/#/login`

### 4. 浏览器验证要看真实页面效果

- 验证不应只停留在“没有报错”，还要确认页面元素真实渲染出来。
- 对角色页面至少确认：
  - 能登录
  - 能跳到首页
  - 核心 Tab 能切换
  - 关键字段能显示
  - 关键按钮可点击

### 5. 功能联调要分层验证

- 页面问题先验证页面是否渲染。
- 上传问题先验证接口返回 URL，再验证磁盘是否真正落盘。
- 支付问题先验证配置是否生效，再验证支付发起、回跳/回调、状态同步。
- 不要把多个问题混在一次排查里。

### 6. 外部服务优先准备本地替代方案

- 对象存储未打通时，优先切本地文件持久化。
- 支付未完全打通时，先保证配置真实可读、支付入口可发起、状态可同步。
- 异步通知如果依赖公网回调地址，本地环境可先验证同步返回和状态查询链路。

### 7. 配置项必须“可改且生效”

- 不能只把参数写进 `application.yml`，还要确认代码运行时真实读取了这些配置。
- 支付、存储、数据库等关键配置必须避免“表面可配、实际写死”。

### 8. 课堂展示导向下的联调标准

以下条件满足即可视为“当前阶段可用于课堂展示”：

- 前后端能独立启动成功
- 四类角色至少有可进入的页面
- 主流程可以从头演示到尾
- 关键资源可读写
- 支付、上传、订单状态等关键节点有可说明的结果反馈
- 出现失败时有明确提示，而不是页面崩溃或无响应

## 协作建议

- 每次开始新开发前，先对照 `待完善功能.md` 确认所属优先级和模块。
- 每完成一个模块，至少补一次“启动 + 页面/接口最小验证”。
- 若一个问题会阻塞主流程演示，应优先处理阻塞项。
- 若一个优化只提升工程完整度但不提升课堂展示效果，应延后处理。
- 若新增了阶段文档、更新了验收范围、调整了路线图或修改了关键结论，应先判断它影响的是“原则”“进度”还是“Phase 1 验收内容”，再按职责范围更新对应文档。

## 一句话原则

这个项目当前阶段的核心目标不是“做成生产级外卖平台”，而是“做成一套能支撑课堂展示、角色完整、流程闭环、可稳定演示的外卖系统”。
